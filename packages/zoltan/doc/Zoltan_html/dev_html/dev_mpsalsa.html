Appendix: Testing in MPSalsa

The Zoltan library was tested in MPSalsa, a 3D unstructured finite element code for simulating chemically reacting flow [MPSalsa User's Guide]. Currently, calls to Zoltan are included in the steady-state solver. A test problem is solved using the original decomposition provided by nem_slice. A new decomposition is then computed through a call to Zoltan (call_load_balancing function in MPSalsa's rf/rf_dlb.c). The simulation is run again using the new decomposition and the solution from the previous solve as the initial guess for the new solve. Thus, in MPSalsa's output, the initial values for the new solve should correspond with the final values of the previous solve. This condition can be checked coarsely by comparing the minimum, maximum and average solution values of the old solve's final solution with the new solve's initial guess; it can be checked more carefully by using MPSalsa's debug level of 7 and comparing each solution vector entry for the old and new solve. Because the initial guess for the new solve is the steady-state solution, only one (or perhaps, two) Newton steps should be required in the new solve; any greater number probably indicates a problem with migration of the solution vector.

After the second solve, Zoltan is invoked again after registering an ZOLTAN_OBJ_WEIGHT_FN query function. Thus, load-balancing is performed this time using object weights. In the current implementation, nodal weights are determined arbitrarily by the nodal coordinates. To see the values to which the weights are set, one should look at assignments to the Weight field of the Node_Struct data structure defined in rf/rf_node_const.h. These weights can be set to any value desired by the user. More intelligent weighting schemes that, say, take into account nodal fill times or number of non-zeros on the node's row of the matrix, can (and probably should) be implemented.

The load-balancing query functions and all data migration functions are implemented in rf/rf_dlb.c and rf/rf_dlb_const.h. Data types for global and local object IDs are defined in rf/rf_dzoltan_user_data.h; this file should be copied to lb/zoltan_user_data.h before compiling Zoltan. Global IDs are integers representing global node or element numbers. Local IDs are pointers to nodes or elements. Calls to load-balancing library functions are currently in rf/rf_salsa.c (for initialization) and rf/rf_solve.c (to register weight functions). All load-balancing code is enclosed within "#ifdef DLB / #endif" controls to prevent interference between load-balancing testing and other project development.

A working distribution of MPSalsa can be obtained from CVS using the following command:

cvs checkout MPSalsa
Minor changes to the Makefiles are necessary to enable the DLB flag and set paths to the load-balancing interface files. In the top-level MPSalsa Makefile, the standard MPSalsa distribution from CVS has
SUBVPATH=../rf ../el ../md ../ps ../pe ../ds
SUBINCPATH=-I../rf -I../el -I../md -I../ps -I../pe -I../ds
Assuming Zoltan's CVS distribution is in a directory named DLB in the same directory as MPSalsa's CVS distribution, these lines should be changed to
SUBVPATH=../rf ../el ../md ../ps ../pe ../ds ../../DLB/lb
SUBINCPATH=-I../rf -I../el -I../md -I../ps -I../pe -I../ds -I../../DLB/zz
Commented versions of these lines are included in the top-level Makefile.

In lower-level MPSalsa Makefiles, the DLB flags should be activated by removing the comment character from lines defining DLB_DEF and DLB_LIB. For example, in MPSalsa's Obj_smos/Makefile, the following lines should be included:

DLB_DEF = -DDLB
DLB_LIB =-L../../DLB/Obj_smos -lzoltan -L/Net/mp/mpi/new/lib/paragon/sunmos-nx -lmpi
An MPSalsa executable is then built using gmake and specifying the target architecture; e.g.,
gmake smos
builds the MPSalsa executable for the Intel Paragon sipapu.

Unfortunately, the exclusion of the load-balancing code from the standard compilation of MPSalsa makes testing by other MPSalsa users unlikely; perhaps a load-balancing developer can occasionally run MPSalsa's parallel software test suite, comparing the text output files, to ensure that changes to MPSalsa haven't broken the data migration routines. The more elegant solution of automatically running a parallel version of MPSalsa with the load-balancing enabled would require significantly changing MPSalsa's dtest utility, a most unpleasant task.

The load-balancing method and parameters for the method can be specified in the MPSalsa input file in the User-Function section. An example user-function section for RCB load balancing is included in the figure below. The Function Name specification states that the LB_set_up function receives three FN_DATA data lines. The first data line contains a string that specifies the name of the load-balancing method to be used in the call to Zoltan_LB_Set_Method. The string NONE is a valid option on this line if no balancing is desired.

The second data line contains the floating-point parameters to be passed to Zoltan in Zoltan_LB_Set_Method. These parameters must be listed in the order that the load-balancing method would expect them. If a FLOAT FN_DATA line is not included, the default load-balancing parameters are used.

The third data line contains two integers. The first integer specifies the amount of debugging information MPSalsa prints during load balancing and data migration. Higher values of this integer produce greater amounts of debugging output. The second integer enables MPSalsa to test the use of different MPI communicators in Zoltan. If the integer is 0, all processors participate in load balancing. If the integer is 1, only processors with rank less than or equal to N/2 participate, where N is the number of processors. If the integer is 2, two separate instances of load balancing occur simultaneously, one among the lower N/2 processors and the other among the remaining processors. If an INT FN_DATA line is not included, the default debugging level (zero) and MPI communicator (MPI_COMM_WORLD) are used.

     Data Specification for User's Functions
------------------------------------------------------------
Number of functions to pass data to = 1
Function Name = LB_set_up  3
FN_DATA = STRING RCB
FN_DATA = FLOAT  1.0 0.0
FN_DATA = INT 0 0
User-function section of the MPSalsa input file to select RCB load balancing.
 

Some example MPSalsa problems that use the load balancing library are in the directory

/Net/pecos/rf_crew/people/kddevin/ex/zz
MPSalsa input files are typically named in.* or input.*. Refer to the MPSalsa User's Guide for more details on running MPSalsa.

To visualize nodal weights or processor assignments, an auxiliary MPSalsa output file must be created. Such a file can be built using the add_var utility:

add_var -n 2 -l Weight, Processor -o exo_file-out-aux.exoII exo_file.exoII
where "exo_file" corresponds to the ExodusII mesh file specified in MPSalsa input file. (The command " add_var -h" provides help for using add_var.) An Auxiliary Output Specification must then be added to the MPSalsa input file; this section must precede the Parallel I/O Section. An example of the Auxiliary Output Specification for nodal weights and nodal processor assignments is shown in the figure below. The name of the auxiliary output file should be the name of the file created by the add_var utility. The number of auxiliary nodal variables is two: the nodal weights and the nodal processor assignments. The functions f_dlb_weight and f_dlb_processor are MPSalsa functions that return the nodal weights and processor assignments, respectively. After MPSalsa's execution, the auxiliary output file can be viewed with AVS, Mustafa, blot, or any graphics utility that reads ExodusII files.
 
 
-------------------------------
Auxiliary Output Specifications
-------------------------------
Auxiliary output file = exo_file-out-aux.exoII
Number of auxiliary nodal variables = 2
Auxiliary nodal variable and function name(s):
AUX_OUT = Weight FUNCTION = f_dlb_weight
AUX_OUT = Processor FUNCTION = f_dlb_processor
Auxiliary Output section of the MPSalsa input file to create an auxiliary ExodusII file for nodal weights and nodal processor assignments.
 

Data migration in MPSalsa does not yet work for problems where mesh refinement is used, as parent and child elemental information is not yet migrated.
